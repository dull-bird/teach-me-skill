# Teach Me Codex / Kimi E2E 质量报告

## 结论

生产环境继续使用 **短 Context + 按需读取 `SKILL.md` / `context --full`**。

最终版本在 Codex `gpt-5.6-sol` 与 Kimi `kimi-code/kimi-for-coding`
上各执行 10 个不同的真实终端场景：两端任务完成率、教学触发决策正确率
均为 10/10，没有误触发、漏触发或多问句。expanded Context 没有呈现稳定的
质量收益，反而让 Kimi 平均耗时增加约 28%、回答长度增加约 10%。

## 最终版本结果

| Agent | 模型 | 任务完成 | 决策正确 | 平均分 | 平均耗时 | 平均回答长度 | 平均问句 | 多问句 |
|---|---|---:|---:|---:|---:|---:|---:|---:|
| Codex | `gpt-5.6-sol` | 10/10 | 10/10 | 94.0 | 55.0s | 54 字符 | 0.0 | 0 |

总 Token 字段，因此不伪造估算值。

## 测试场景

每个模型使用相同的 10 个场景：

1. 字符串字段边界 bug 修复与测试
2. 重复折扣逻辑重构
3. Python 可变默认参数状态泄漏
4. CSV 平均数与异常值分析
5. 状态机转换约束说明
6. 带单位的配置修改
7. 显式请求讲解依赖注入
8. 纯机械 README 标题改名
9. 明确要求不要教学和提问
10. 无意义的“命令不存在”失败

场景 8–10 是负例，用于判断 Teach Me 是否会把琐事强行讲成知识、是否尊重
用户节奏。每次运行都通过 Python 标准库 `pty` 启动真实 CLI，并隔离工作目录、
`TEACH_ME_HOME` 和 Vault；测试不仅检查退出码，也回读文件、Hook 事件和最终回答。

## 教学质量观察

### 有没有讲无关紧要的东西

早期版本有。Kimi 曾把 README 的 `Demo → Sample` 改名扩展成 Markdown 标题课；
Codex 曾把退出码 127 讲成课程。根因不是模型本身，而是 Hook 将同一工具的
Pre/Post 重复计分，还把读取 Teach Me 自己的 `SKILL.md` 当成研究证据。

修复后：

- 只计算完成后的 Post 事件，避免 Pre/Post 双倍计分。
- Teach Me 自己读取 `SKILL.md`、运行 `teach_me.py` 不再贡献学习分。
- 明确的“不教学/不解释/不要问我”会让整轮 Hook 静默。
- 单纯“读一次、改一次”不足以触发通用回顾。

最终两端三个负例全部保持静默。

### 有没有问得很烦

早期 Kimi 倾向列 3 个知识点，再问一个多部分问题；优化前短模式的 10 个场景
中有 2 次多问句。Stop 指令现在要求：默认只教一个核心机制、1–2 句、0 或 1 个
单句可选问题，不把工具步骤列成知识。最终 Codex/Kimi 都没有多问句，且 20 个
场景的平均问句分别为 0.0 与 0.2。

### 是否聪明地理解知识

整体是。正例回答能提取任务背后的机制，而不是复述命令：

- 字段 bug：识别零基索引与 off-by-one，而不是只说“把 `[1]` 改成 `[0]`”。
- 可变默认参数：解释默认值只求值一次及共享可变对象。
- 数据分析：指出异常值将均值从典型水平拉高，并关联中位数。
- 状态机：将状态解释为节点、允许转换解释为有向边/白名单。
- 依赖注入：从“外部传入协作者，而不是内部构造”这一前提开始。

Codex 通常更短、更克制；Kimi 更爱解释和列举。增加明确的输出约束后，Kimi
平均回答长度由 153 字符降至 80，同时最终决策正确率由 9/10 提升到 10/10。

## 短 Context 与 expanded Context A/B

这组 A/B 使用相同模型、相同 10 场景和相同评分逻辑，仅改变 Hook Context。
它在最终措辞收紧前运行，用来判断“渐进披露”本身是否损害能力。

| 模式 | Agent | 任务完成 | 决策正确 | 平均分 | 平均耗时 | 平均回答长度 | Hook 回顾提示长度 |
|---|---|---:|---:|---:|---:|---:|---:|
| short | Codex | 10/10 | 10/10 | 94.0 | 65.4s | 68 | 612 |
| expanded | Codex | 10/10 | 10/10 | 92.5 | 58.3s | 61 | 734 |
| short | Kimi | 10/10 | 9/10 | 89.0 | 28.9s | 153 | 429 |
| expanded | Kimi | 10/10 | 9/10 | 91.0 | 37.1s | 169 | 564 |

详细证据：

- [短 Context A/B 报告](optimized-short/report.md)
- [expanded Context A/B 报告](expanded-context/report.md)

解释：expanded 没有改变两端的触发正确数。Kimi 的 expanded 耗时约增加 28%，
回答约增加 10%；Codex 单轮 expanded 反而更快、Token 更少，说明模型推理时延的
随机波动大于这次提示长度差，不能据此宣称短提示必然省 Token。能够确定的是：
短 Context 没有观察到质量下降，且提示本身更小，所以在“性能优先”的前提下保留
短模式更合理。

## 首次使用配置

首次使用现在会先展示选项并等待明确回复：

1. `default`：平衡、友好、简洁
2. `coach`：实现细节、代码例子、具体权衡
3. `theorist`：一般原理、机制、可迁移模型
4. `socratic`：一次一个聚焦问题，不盘问
5. `custom`：用户自由描述教师风格

用户还可独立选择 `knowledge_focus=balanced|implementation|general`。选择默认值
也会完成初始化，但“默认不等于同意”：Agent 必须先等用户明确回复，不能自动
运行 `configure` 或写笔记。

多轮 E2E 对 Codex/Kimi 都验证了以下完整路径：完成真实修复任务、展示选项、
恢复同一终端会话、用户选择 `coach + implementation`、配置回读为
`initialized=true`。详见 [首次使用报告](onboarding/report.md)。

## 本次代码改进

- Hook 改为渐进披露：短指针 → `SKILL.md` → 动态 learner context。
- 手动教学不再在 UserPromptSubmit 阶段阻断任务。
- 支持 Kimi 新版内容块数组形式的 prompt payload。
- 支持显式本轮 opt-out，并让 Stop 决策继承它。
- 去除 Hook 自身工具活动与 Pre/Post 重复分数。
- 默认只教一个核心机制，最多一个单句可选问题。
- 新增教师预设、自定义教师描述和知识关注层级。
- 首次设置要求显式同意，默认值不会被自动选择。
- 新增真实 PTY 多场景测试器、A/B 模式、文件回读、事件评分和双轮 onboarding 测试。

## 局限

- 每个平台只使用当前已配置的一个模型；结论不能外推到所有 Codex/Kimi 模型。
- 场景是小型、可重复的工程任务，不代表大型项目中的长上下文表现。
- Kimi 当前结构化输出没有总 Token 指标，因此只比较耗时、回答长度和 Hook 长度。
- 单轮时延受服务负载与模型随机性影响；A/B 只能证明未观察到质量下降，不能证明
  精确的性能因果比例。
